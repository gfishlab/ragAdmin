# 子代理路由规则

## 目标

在复杂任务中引入最小必要的角色分工，让任务拆解、实施和验证保持清晰边界。

## 角色

- `planner`
- `executor`
- `verifier`

## 默认路由

### 简单任务

适用特征：
- 单文件或单模块小改动
- 验收标准清晰
- 跨模块依赖很少

默认动作：
- 主 Agent 直接执行
- 完成后按 `verifier` 视角自检

### 中等任务

适用特征：
- 涉及两个及以上模块
- 存在边界判断
- 需要兼顾文档与实现一致性

默认动作：
- 先由 `planner` 梳理边界与验收点
- 再由 `executor` 实施改动
- 最后由 `verifier` 汇总结论

### 复杂任务

适用特征：
- 跨后端、前端、数据库或流程机制
- 影响范围广
- 需要并行收集信息或分阶段验证

默认动作：
- `planner` 先形成拆解方案
- 按边界分配一个或多个 `executor`
- 由 `verifier` 做统一验收与风险回看

## 并行执行判断

每次任务开始时，根据具体任务结构判断是否适合并行执行：

### 适合并行的特征

- 任务可拆解为多个独立子任务，子任务之间无写冲突
- 多个文件的改动互不依赖（如新建多个独立类）
- 存在明确的依赖图，DAG 中同一层级的任务可同时推进
- 预计总耗时较长，并行可显著缩短执行时间

### 适合线性的特征

- 子任务之间有严格的先后依赖（Step B 需要 Step A 的产物）
- 改动文件有重叠或交叉引用，并行容易产生冲突
- 任务较轻量，并行调度的开销反而更大

### 并行方式

- 使用 `TeamCreate` 创建团队，通过 `TaskCreate` 拆分任务
- 用 `Agent` 工具（设置 `team_name`）启动多个 teammate 并行工作
- 每个 teammate 分配独立的文件或模块，避免编辑冲突
- 主 Agent 负责任务分配、依赖管理和最终集成

### 判断时机

- 在 plan 完成后、进入实施前做一次判断
- 不强制要求所有任务都并行，也不强制所有任务都线性
- 由主 Agent 根据任务依赖图自行决定，不需要用户额外指示

## 角色协同提示

- `planner` 重点输出边界、风险、验收项
- `executor` 重点输出实现结果和显式风险
- `verifier` 重点输出验证结论、未覆盖项和建议沉淀内容
