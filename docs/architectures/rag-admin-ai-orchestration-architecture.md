# ragAdmin AI 编排架构设计

## 1. 文档定位

本文档用于收敛 `ragAdmin` 在 AI 问答编排侧的正式技术设计，统一回答以下问题：

- 现实世界问答、知识库问答与工具增强应如何分层
- Prompt 模板、结构化输出、联网搜索应分别承担什么职责
- 哪些结果适合结构化输出，哪些不适合
- 联网增强为什么默认是可控补充能力，而不是核心事实源
- 后端应如何组织 Prompt 模板、编排链路、回退策略与测试方式

当前阶段如果局部实现与本文档冲突，应优先更新本文档，再调整实现。

## 2. 设计边界

本文档只覆盖 `ragAdmin` 的 AI 编排层，不覆盖以下内容：

- 基础系统总体架构，见 `rag-admin-architecture.md`
- API 契约，见 `rag-admin-api-design.md`
- 数据库结构，见 `rag-admin-schema-v1.sql`
- 会话长期 / 短期记忆分层，见 `rag-admin-chat-memory-architecture.md`

## 3. 总体原则

### 3.1 知识库是主事实源

内部制度、流程、SOP、组织口径、权限规则、客户资料等问题，主事实源必须始终是内部知识库，而不是互联网。

### 3.2 联网能力是可控增强

联网增强主要用于知识库之外且强时效的问题，例如天气、新闻、政策动态、行业公开信息。默认关闭，由用户或管理员显式开启。

### 3.3 Prompt 不承担全部职责

Prompt 的职责应收敛为：

- 表达回答原则
- 消费已准备好的上下文
- 约束输出边界与格式

Prompt 不应承担：

- 所有现实世界事实判断
- 工具能力路由
- 隐式时间和地点解释

### 3.4 编排优于补丁式修 Prompt

不要采用“遇到一个 case 补一句 Prompt”的方式修系统问题。稳定性优先来自输入归一化、能力路由、上下文结构化和回退策略。

## 4. 编排分层模型

AI 问答编排至少分为四层：

### 4.1 输入归一化层

负责把用户自然语言中的隐式上下文显式化、结构化，例如：

- 将“今天 / 明天 / 昨天”转换为绝对日期
- 将“本周 / 下周 / 本月底”转换为明确时间区间
- 将“这里 / 当地 / 附近”绑定为显式地区
- 将“最新 / 刚刚 / 当前”标记为强时效查询

这是一种通用上下文标准化能力，不应退化成天气特判。

### 4.2 能力路由层

负责决定问题应该走什么能力，而不是默认都交给大模型自由发挥。

推荐路由如下：

- 内部知识、制度、流程、文档事实：优先知识库检索
- 天气、股价、汇率、地图、日历、航班、快递等强时效强约束问题：优先专用 tool / MCP / 外部 API
- 行业动态、政策新闻、公开资讯：可优先走联网搜索增强
- 企业内部业务查询：优先企业内部 `MCP`

### 4.3 上下文编排层

负责把知识库检索结果、联网摘要、工具返回结果、系统时间和场景参数拼装为模型可消费上下文。

必须保证：

- 上下文来源可区分
- 知识库事实与联网结果不混淆
- 当前日期、时区、地区等现实世界约束显式传入模型

### 4.4 输出控制层

负责决定结果以自然语言、结构化对象或混合模式返回，并承担回退与异常兼容逻辑。

## 5. 联网增强设计

### 5.1 产品定位

联网增强在 `ragAdmin` 中的定位如下：

- 内部知识库问答：以知识库为主，联网只做补充
- 通用问答：允许按权限开启联网增强
- 默认关闭联网，由用户或管理员显式开启

### 5.2 Provider 选择结论

当前阶段优先接入 `Tavily`，原因如下：

- 面向 AI Agent / RAG 场景设计，返回结果更适合直接进入编排层
- 接入成本低，适合先落最小可用版本
- 计费模型清晰，便于预算控制
- 当前已有 `WebSearchProvider` 抽象，不需要一开始就做复杂的多搜索源平台

### 5.3 技术实现约束

- 联网搜索适配统一放在 `infra.search`
- 保持 `WebSearchProvider` 抽象
- 使用配置项管理 `apiKey`、`baseUrl`、`searchDepth`、超时、结果条数
- 搜索失败时必须可降级，不能中断主问答链路
- 回答中显式区分“知识库引用”和“联网结果”
- 审计日志记录是否开启联网、搜索词、调用耗时、结果条数

## 6. Prompt 模板治理

### 6.1 模板组织原则

- Prompt 模板文件统一放到 `resources/prompts`
- 统一使用 `.st` 模板文件
- 模板只承载文本，不承载复杂流程控制
- 统一由 `PromptTemplateService` 渲染

### 6.2 渲染与注入原则

- 通过 `@Value` 注入 `Resource`
- 渲染缺变量即失败，避免静默降级
- 模板命名与业务场景保持一致，避免散落和重复语义

### 6.3 模板职责边界

Prompt 模板应只消费已准备好的上下文，不直接发明上下文。现实世界时间、地区、联网摘要、检索结果等必须先由编排层准备好，再交给模板组织输出。

### 6.4 何时必须上专用 tool / MCP

以下场景不应只靠 Prompt 或联网摘要硬撑：

- 强时效、强精度、强约束的外部事实查询
- 企业内部系统实时业务查询
- 需要明确参数和可审计结果的工具调用

## 7. 结构化输出设计

### 7.1 适用原则

结构化输出适合“结果还要被程序继续消费”的场景，不适合开放式自然语言正文。

### 7.2 当前适合的场景

- 会话摘要生成
- 意图识别
- 检索规划
- 工具参数生成
- 回答后处理元数据

### 7.3 当前不适合的场景

- 面向用户展示的普通正文回答
- 逐 token 流式聊天正文
- 长篇自由创作类回答
- schema 仍不稳定的探索期场景

### 7.4 推荐输出模式

- 非流式结构化输出：用于摘要、意图、规划、工具参数
- 流式自然语言输出：用于问答正文
- 混合模式：正文流式返回，结束时补一个结构化尾事件

### 7.5 schema 设计原则

- 字段越少越稳定
- 枚举优先于自由文本
- schema 必须由服务端控制
- 必须设计回退策略，不能让前端承担结构化纠错职责

## 8. 当前推荐落点

### 8.1 已确认的架构落点

- 会话摘要：优先结构化输出
- 检索意图与检索规划：优先结构化输出
- 正文回答：优先自然语言流式输出
- 联网增强：只作为可控补充链路
- 企业内部工具：优先走内部 `MCP`

### 8.2 当前不建议立即改造的链路

- 将普通聊天正文强制改成结构化输出
- 逐 token 输出 JSON
- 为了单一 case 不断堆 Prompt 分支

## 9. 测试与回退

### 9.1 测试要求

- Prompt 模板渲染缺变量必须失败
- 编排层测试要覆盖时间归一化、联网开关、来源隔离、降级路径
- 结构化输出测试要覆盖 schema 校验、字段缺失与回退逻辑

### 9.2 回退要求

- 联网失败时回退到无联网主链路
- 结构化输出失败时回退到可接受的自然语言或默认对象
- 工具调用失败时保留显式失败信号和审计记录

## 10. 最终原则

一句话总结：

`ragAdmin` 的 AI 稳定性优先来自输入归一化、能力路由、来源隔离、模板治理与输出控制，而不是持续堆叠 Prompt 补丁。
