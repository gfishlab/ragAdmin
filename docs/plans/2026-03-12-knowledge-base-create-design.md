# 知识库创建页设计说明

## 1. 目标

本次实现目标是在当前管理端前端基线之上，补齐“知识库创建”最小可用闭环：

- 提供独立的知识库创建页面
- 创建成功后返回知识库列表页
- 列表页自动刷新并提示创建成功
- 模型下拉优先走真实接口，失败时退化为平台默认模型兜底

本轮不追求一次性做完编辑、详情、文档管理等后续能力，只完成一个可运行、可评审、可继续迭代的垂直切片。

## 2. 方案选择

本次采用“独立路由页 + 列表回跳联动”方案，而不是在列表页里弹出表单弹层。

选择该方案的原因：

- 符合当前仓库“小而完整垂直切片”的实现原则
- 页面边界清晰，便于后续复用到“编辑知识库”场景
- 提交流程、返回逻辑、成功提示都更容易在路由层表达

路由设计如下：

- 列表页：`/knowledge-bases`
- 创建页：`/knowledge-bases/create`

## 3. 页面结构

创建页保持单页单表单，不提前拆复杂区域。字段对齐当前接口设计文档：

- `kbCode`
- `kbName`
- `description`
- `embeddingModelId`
- `chatModelId`
- `retrieveTopK`
- `rerankEnabled`
- `status`

页面交互约束如下：

- `kbCode`、`kbName`、`retrieveTopK`、`status` 为必填
- `description` 为可选
- 模型字段允许为空，为空时表示使用平台默认模型兜底
- 页面提供“取消返回”和“提交创建”两个主要操作

## 4. 数据流设计

### 4.1 创建知识库

前端新增 `createKnowledgeBase` API 方法，请求体严格对齐 `POST /api/admin/knowledge-bases`。

提交流程：

1. 用户填写表单
2. 前端执行基础校验
3. 调用创建接口
4. 创建成功后跳回列表页，并通过路由查询参数携带一次性成功标记
5. 列表页识别成功标记后刷新列表并提示成功
6. 刷新后清理成功标记，避免重复提示

### 4.2 模型下拉兼容策略

模型下拉优先请求真实模型接口，并按能力拆分为：

- 聊天模型
- Embedding 模型

当模型接口不可用、返回异常或无数据时，页面进入降级模式：

- 不阻塞知识库创建
- 模型字段保持可为空
- 页面提示“当前将使用平台默认模型”
- 不把模型接口失败升级为页面致命错误

这样可以避免模型管理接口尚未完全稳定时，影响知识库创建主流程。

## 5. 错误处理

错误处理分两层：

- 表单校验错误：直接在字段级提示，阻止提交
- 接口调用错误：在页面级给出明确失败提示，并保留用户输入

同时需要具备以下交互约束：

- 提交时按钮进入 loading，防止重复提交
- 接口失败时不清空表单
- 取消返回不做复杂草稿机制，本轮不引入额外状态存储

## 6. 组件与复用边界

本轮只做必要复用，不做过早抽象。

建议拆分边界：

- 列表页继续保留为 `KnowledgeBaseListView`
- 新增创建页 `KnowledgeBaseCreateView`
- 抽取一个轻量知识库表单组件，供创建页使用，后续编辑页可复用
- API 层新增创建接口与模型列表读取接口

不在本轮引入：

- 创建 / 编辑双场景统一容器
- 全局模型选项缓存仓库
- 草稿保存
- 详情页与编辑页

## 7. 验证范围

本轮至少完成以下验证：

- 创建页可正常访问
- 模型接口成功时可显示真实下拉
- 模型接口失败时可降级并继续创建
- 创建成功后能回到列表页并自动刷新
- 前端项目可通过 `npm run build`
