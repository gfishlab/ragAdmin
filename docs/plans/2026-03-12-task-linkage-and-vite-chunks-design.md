# 任务联动与 Vite 拆包优化设计说明

## 1. 目标

本次实现目标包含两部分：

- 在任务详情页补充更明确的业务关联信息展示
- 通过 `Vite manualChunks` 对前端公共大依赖做稳定拆包，消除当前包体告警

本轮重点是以最小风险方式同时提升“任务可读性”和“前端构建产物可维护性”。

## 2. 方案选择

本次采用“任务详情保守联动 + 最小风险拆包”方案。

选择原因如下：

- 任务详情当前已有 `taskType` 与 `bizId`，可以先做保守增强
- `manualChunks` 是处理当前告警的最小改动路径
- 不需要为本轮目标引入大规模组件重构

## 3. 任务联动设计

任务详情联动遵循“能确定再跳，不能确定就只展示”的原则。

当前处理策略：

- 如果 `taskType` 属于文档解析类任务，则将 `bizId` 视为关联文档 ID 展示
- 当前若没有独立文档详情页，则先只展示关联文档信息和说明文案
- 不强行根据 `bizId` 反推知识库或其他业务对象

这样可以保证前端不会做出错误跳转。

## 4. 拆包设计

在 `vite.config.ts` 中通过 `build.rollupOptions.output.manualChunks` 进行稳定拆包。

拆包策略：

- `framework`：`vue`、`vue-router`、`pinia`
- `ui`：`element-plus`、`@element-plus/icons-vue`
- `http`：`axios`

目标：

- 拆分当前超大的主入口 chunk
- 降低单个 chunk 体积
- 优先消除当前构建时的 500 kB 告警

## 5. 本轮边界

本轮包含：

- 任务详情业务关联信息展示
- `vite.config.ts` 的 `manualChunks` 配置
- 重新构建并检查告警结果

本轮不包含：

- 文档详情页
- 强业务跳转闭环
- 组件层大规模改造
- 更激进的依赖引入重构

## 6. 验证范围

本轮至少完成以下验证：

- 任务详情页能更清楚展示业务关联信息
- 前端项目可通过 `npm run build`
- 构建后的主包被拆分，不再只剩一个超大入口包
- 若可能，消除当前 Vite 500 kB 告警

## 7. 后续建议

在本轮完成后，建议继续按以下顺序推进：

1. 文档详情页
2. 任务与文档详情强联动
3. 更细粒度的前端包体优化
