# 知识库模型边界设计

## 背景

当前系统已经把默认聊天模型收敛到后台模型管理，但知识库仍然保留 `chatModelId`，同时向量链路还允许在未绑定 `embeddingModelId` 时走配置文件兜底。这会造成两个问题：

- 业务配置来源分叉，知识库、模型管理和 `application.yml` 都可能影响运行结果
- 向量化链路缺少明确绑定，文档切片、向量索引和检索结果难以稳定追踪

## 目标

- 知识库必须绑定一个向量模型
- 知识库不再配置聊天模型
- 平台不再提供默认向量模型兜底
- 问答生成统一走平台默认聊天模型，前台会话或单次请求仍可显式覆盖

## 方案

### 1. 知识库配置边界

- `embeddingModelId` 保留在知识库中，并调整为必填
- `chatModelId` 从知识库 API 契约中移除
- 数据库历史列 `kb_knowledge_base.chat_model_id` 暂不直接删除，只保留兼容意义，运行时代码不再读取

### 2. 运行时模型解析

- 文档解析、切片向量化、检索都只读取知识库绑定的 `embeddingModelId`
- `ModelService.resolveEmbeddingModelDescriptor(null)` 直接报错，不再回退配置文件
- 后台问答统一使用平台默认聊天模型
- 前台问答统一采用“请求显式模型 > 会话显式模型 > 平台默认聊天模型”

### 3. 管理台交互

- 知识库表单只保留向量模型选择，且必须选择
- 页面显式说明“回答模型策略”为平台默认聊天模型
- 模型列表加载失败时，新建知识库不可继续提交；编辑页允许保留当前已绑定向量模型

### 4. 数据演进

- 新增 Flyway 迁移，对 `kb_knowledge_base.embedding_model_id` 增加未来写入必须非空的约束
- 不直接删除 `chat_model_id`
- 同步更新 API、架构、SQL 草案与验收文档

## 风险与控制

- 历史知识库若未绑定向量模型，统计页会明确暴露为未绑定状态，避免静默回退
- 旧代码若仍依赖知识库 `chatModelId`，测试会直接失败，便于一次性收口
- 配置文件中的旧 `default-embedding-model` 键移除后，可避免误以为系统仍支持默认向量模型
