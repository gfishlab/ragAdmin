# 知识库文档上传入口设计说明

## 1. 目标

本次实现目标是在现有知识库详情页与文档列表浏览能力之上，补齐“单文件上传接入知识库”最小闭环：

- 在知识库详情页提供上传文档入口
- 获取上传凭证
- 通过预签名地址直传对象存储
- 新增文档记录并挂到当前知识库下
- 上传成功后刷新当前文档列表

本轮重点是把“知识库 -> 文档接入”真正跑通，不把解析、版本管理等后续能力一起卷入。

## 2. 方案选择

本次采用“详情页触发上传弹窗”方案。

选择原因如下：

- 不破坏当前详情页主体结构
- 上传交互集中，用户上下文清晰
- 后续若扩展批量上传、解析触发或更多校验，弹窗仍有承载空间

不采用独立上传页，是因为当前阶段实现成本偏高；不采用详情页内联上传区，是为了避免详情页信息区过重。

## 3. 交互设计

上传入口放在知识库详情页文档列表区右上角，作为主操作按钮存在。

点击后打开上传弹窗，弹窗内只服务当前知识库，不再要求用户重新选择知识库。

弹窗内容包括：

- 文件选择区
- 文件基础信息展示
- 上传状态说明
- 取消 / 开始上传按钮

本轮约束如下：

- 只支持单文件上传
- 不做拖拽上传
- 不做批量队列
- 不做上传后自动触发解析

## 4. 数据流设计

上传链路严格分为三步：

### 4.1 获取上传凭证

调用 `POST /api/admin/files/upload-url`，提交：

- `fileName`
- `contentType`
- `bizType: KB_DOCUMENT`

返回：

- `bucket`
- `objectKey`
- `uploadUrl`

### 4.2 对象存储直传

前端使用预签名地址直接上传原始文件，不通过统一业务 API 客户端处理，避免鉴权头或响应解包逻辑干扰。

### 4.3 新增文档记录

上传成功后，调用 `POST /api/admin/knowledge-bases/{kbId}/documents`，提交：

- `docName`
- `docType`
- `storageBucket`
- `storageObjectKey`

成功后关闭弹窗，并刷新当前知识库文档列表。

## 5. 文档类型处理

首版采用前端基于文件扩展名的简单映射：

- `pdf -> PDF`
- `doc` / `docx -> DOC`
- `md -> MARKDOWN`
- `txt -> TEXT`

如果无法识别，则退化为 `UNKNOWN`。

如果后端后续对 `docType` 枚举收紧，再按后端契约同步收口。

## 6. 错误处理

错误处理需要区分阶段，避免用户不知道失败发生在哪一步：

- 获取上传凭证失败：提示“获取上传凭证失败”
- 对象存储上传失败：提示“文件上传失败”
- 新增文档记录失败：提示“文件已上传，但文档登记失败”

同时要求：

- 上传过程中禁止重复点击
- 失败后保留当前已选文件，允许直接重试
- 上传成功后清空弹窗状态

## 7. 本轮边界

本轮包含：

- 上传按钮
- 上传弹窗
- 获取上传凭证
- 对象存储直传
- 新增文档记录
- 上传成功后刷新列表

本轮不包含：

- 批量上传
- 拖拽上传
- 进度条精细展示
- 自动触发解析
- 文档版本管理
- 失败补偿后台机制

## 8. 验证范围

本轮至少完成以下验证：

- 能从知识库详情页打开上传弹窗
- 能成功获取上传凭证
- 能完成对象存储直传
- 能成功新增文档记录并刷新列表
- 任一阶段失败时能给出明确错误提示
- 前端项目可通过 `npm run build`

## 9. 后续建议

在上传闭环完成后，建议继续按以下顺序推进：

1. 触发文档解析
2. 文档详情与版本信息
3. 解析任务状态联动
