# 前端一期阶段计划与执行记录

## 1. 文档目的

本文档用于汇总 2026-03-12 当天前端一期阶段的设计决策与执行记录，统一替代当天按小功能拆分产生的多个细粒度 plan 文档。

本阶段文档覆盖的能力范围包括：

- 知识库创建、编辑、详情
- 知识库文档列表、上传入口、解析触发
- 任务监控、任务重试、任务详情
- 任务与文档的保守联动
- `Vite` 前端拆包优化

后续同类小功能不再单独新增 plan 文档，默认优先追加到阶段文档中。

## 2. 阶段目标

本阶段目标是把管理端前端从“能登录、能看列表”推进到“能围绕知识库、文档、任务形成基础操作闭环”。

本阶段完成后，应至少具备以下能力：

- 知识库的创建、编辑、详情浏览
- 文档接入知识库并触发解析
- 基础任务可观察与失败任务重试
- 任务、知识库、文档三条线开始建立关联

## 3. 范围与边界

### 3.1 本阶段包含

- 登录与基础后台布局
- 知识库列表页
- 知识库创建页
- 知识库编辑页
- 知识库详情页
- 知识库文档列表
- 单文件上传接入知识库
- 手动触发文档解析
- 任务监控页
- 失败任务重试
- 任务详情页
- 任务详情中的保守业务关联信息展示
- `Vite` 基础拆包优化

### 3.2 本阶段不包含

- 文档详情页
- 文档切片浏览
- 文档版本激活与新增版本
- 任务时间线
- 任务详情中的强业务跳转闭环
- 更激进的前端包体优化

## 4. 关键设计决策

### 4.1 路由结构

当前前端采用独立页面路由，而不是弹窗堆叠式工作流：

- `/knowledge-bases`
- `/knowledge-bases/create`
- `/knowledge-bases/:id`
- `/knowledge-bases/:id/edit`
- `/tasks`
- `/tasks/:id`

该决策的原因是：

- 页面边界清晰
- 后续扩展详情、编辑、版本、切片、任务联动更自然
- 不会让列表页或详情页因为功能累加迅速失控

### 4.2 模型选择策略

知识库创建与编辑中的聊天模型、Embedding 模型采用“双模式兼容”：

- 优先读取真实模型列表
- 接口失败时退化为平台默认模型兜底

设计目标是保证知识库主流程不被模型管理接口拖死。

### 4.3 文档接入策略

文档接入采用三段式闭环：

1. 获取上传凭证
2. 预签名地址直传对象存储
3. 新增文档记录

此处前端不使用统一业务 `http` 客户端上传文件，而是用独立 `fetch` 完成对象存储直传，避免鉴权和统一响应解包干扰预签名上传。

### 4.4 文档解析触发策略

文档解析触发放在知识库详情页文档列表中，以行级按钮发起：

- 仅 `PENDING` / `FAILED` 允许触发
- 只锁当前行，不阻塞整个表格
- 成功或失败后刷新文档列表，以后端真实状态为准

### 4.5 任务监控与治理策略

任务页先做最小可观察性，再逐步引入治理动作：

- 先有列表、筛选、分页、刷新
- 再补失败任务重试
- 再补任务详情

这样可以保证每一轮都有真实用户价值，而不是一次性堆出大而空的“任务中心”。

### 4.6 任务联动策略

任务详情中的业务关联遵循“能确定再展示”的原则：

- 当前能明确识别 `DOCUMENT_PARSE` 类型时，保守展示其关联文档 ID
- 不在前端强行推断知识库或其他对象

该决策是为了避免后端字段尚未稳定时出现错误跳转。

### 4.7 包体优化策略

`Vite` 构建采取“最小风险拆包”而不是激进重构：

- `framework`：`vue`、`vue-router`、`pinia`
- `ui`：`element-plus`、`@element-plus/icons-vue`
- `http`：`axios`

同时将 `chunkSizeWarningLimit` 调整为与当前依赖规模匹配的阈值，目标是先消除构建告警并保持拆包稳定。

## 5. 各能力实现要点

### 5.1 知识库

- 创建页与编辑页复用同一表单组件
- 列表页支持创建回跳刷新与编辑回跳刷新
- 详情页作为知识库工作台承载文档列表
- 文档列表筛选条属于知识库详情页的局部状态，不影响主体信息区

### 5.2 文档

- 当前只做文档接入与解析触发闭环
- 文档详情页采用独立路由 `/documents/:id`
- 文档详情页首版展示主体信息与版本列表，切片浏览后置
- 文档类型通过文件扩展名做前端保守映射

### 5.3 任务

- 任务监控页提供基础筛选与状态颜色区分
- 失败任务支持重试确认与行级 loading
- 任务详情页完整展示错误信息和基础字段
- 任务详情页状态流转时间线采用固定轨迹展示，不伪造中间节点时间

## 6. 验证基线

本阶段代码变更默认至少满足：

- 页面路由可访问
- 关键交互链可跑通
- `npm run build` 通过
- 若涉及构建优化，应明确检查告警是否消失或收敛

## 7. 后续优先级建议

在本阶段基础上，后续前端优先建议如下：

1. 文档版本列表与激活操作增强
2. 文档切片浏览
3. 任务状态流转时间线
4. 任务与文档 / 知识库详情强联动

## 8. 文档收敛规则

从本文档开始，前端一期内的常规小功能不再单独创建 plan 文档。

默认做法：

- 当前阶段内的小功能设计，追加到本文档对应章节
- 只有跨模块、高风险、长周期专题，才允许单独开新文档
- 若某阶段文档过长，再按“前端二期”“后端一期”这类阶段重新拆分

## 9. 文档详情页补充记录

### 9.1 范围

本轮补齐文档详情基础可读页，覆盖：

- 独立文档详情路由 `/documents/:id`
- 文档主体信息展示
- 文档版本列表展示
- 从知识库详情页文档列表进入文档详情页
- 从任务详情页在可识别文档解析任务时进入关联文档

### 9.2 设计要点

- 文档主体详情失败时使用页面级错误态
- 版本列表失败时使用区块级错误态，不影响主体信息展示
- 文档版本激活采用“确认 + 行级 loading + 刷新详情与版本列表”的写操作闭环
- 任务详情到文档详情的跳转仍遵循“能确定再跳”的原则

## 10. 文档版本激活补充记录

### 10.1 范围

本轮补齐文档版本激活最小写操作，覆盖：

- 版本列表非当前版本的“设为生效版本”按钮
- 点击前确认
- 当前行 loading
- 成功后刷新文档详情与版本列表

### 10.2 设计要点

- 当前生效版本不再显示激活按钮
- 前端不本地手改激活状态，而是以后端刷新结果为准
- 激活失败直接展示后端返回文案

## 11. 文档新增版本补充记录

### 11.1 范围

本轮补齐文档新增版本最小闭环，覆盖：

- 文档详情页“新增版本”按钮
- 新增版本弹窗
- 获取上传凭证
- 对象存储直传
- 调用新增版本接口
- 成功后刷新文档详情与版本列表

### 11.2 设计要点

- 新增版本采用弹窗交互，不把上传表单直接堆在版本列表区
- 上传链路沿用“凭证 -> 直传 -> 业务登记”的三段式
- 当前不自动激活新版本
- 当前不引入版本差异比较与变更说明字段

## 12. 知识库文档筛选补充记录

### 12.1 范围

本轮补齐知识库详情页文档区的基础筛选，覆盖：

- `keyword`
- `parseStatus`
- `enabled`
- 查询与重置
- 与分页联动

### 12.2 设计要点

- 筛选条只作用于文档列表，不进入知识库主体区
- 点击查询时回到第一页
- 点击刷新时保留当前筛选与当前分页
- 文档区失败不影响知识库主体区

## 13. 任务状态流转时间线补充记录

### 13.1 范围

本轮补齐任务详情页的状态流转可视化，覆盖：

- 固定状态轨迹展示
- 当前状态高亮
- 已到达 / 未到达区分
- 无精确时间时的占位文案

### 13.2 设计要点

- 时间线固定为 `WAITING -> RUNNING -> SUCCESS / FAILED / CANCELED`
- 前端只基于当前 `taskStatus`、`createdAt`、`updatedAt` 做保守推导
- 无法确定的中间节点时间统一显示“时间未知”
- 不引入步骤树或动态日志面板
